1. Global Conventions
1.1 Base URL & Versioning
All endpoints in this specification are relative to the base URL below. The production domain and staging domain differ only in subdomain; both use TLS 1.3.
Environment
Base URL
Production
https://csidop.10cs.com.my/api
Staging
https://staging.csidop.10cs.com.my/api
Local development
http://localhost:3000/api
Routes are not version-prefixed in Phase 1–3 (single consumer: the platform's own frontend). If Phase 4 external integrations require a stable contract alongside a breaking change, a /v2/ prefix will be introduced at that point without disturbing Phase 1–3 clients still on un-prefixed routes (SAD §8).
1.2 Authentication
Every endpoint in Sections 2–11 except /api/auth/* requires a valid session. The platform uses HttpOnly, Secure, SameSite=Strict cookies set by the /api/auth/callback endpoint after OIDC exchange. The cookie carries a signed JWT; the API middleware verifies the signature on every request using the IdP's public key set (JWKS endpoint), with no database round-trip.

1.3 Request Format
Content-Type: application/json for all POST, PUT and PATCH request bodies
Date-time values: ISO 8601 with time zone — e.g. 2026-06-19T09:30:00+08:00
Date-only values: ISO 8601 date — e.g. 2026-06-30
Monetary values: numeric string with two decimal places — e.g. "150000.00" — never a float
UUIDs: lowercase hyphenated — e.g. "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
Idempotency: POST endpoints for WO creation, effort log and evidence upload accept an optional Idempotency-Key: <uuid> header; duplicate submissions within 24 h return HTTP 200 with the original response body
1.4 Response Envelope
All successful responses use the same JSON envelope, so the frontend client has a predictable shape regardless of which endpoint is called.
{ "success": true, "data": <resource or array>, "meta": <pagination | null> }
All error responses use a mirror envelope:
{ "success": false, "error": { "code": "<CODE>", "message": "<human-readable>", "errors": [<Zod field errors>] } }
1.5 Pagination
List endpoints use cursor-based pagination (stable on high-velocity tables such as EFFORT_LOG). Offset-based pagination is not supported.

1.8 Field-Level Permissions
Some fields within a response body are visible only to specific roles. Where this applies it is called out in the endpoint's Response section. The pattern is: the API always returns the full record shape to the caller's role; fields the caller's role is not permitted to see are omitted entirely (not returned as null), so the frontend can test for the presence of a key rather than checking a permission flag.
1.9 Scope Enforcement Reminder
Row-level scope filtering (PRD §6; SAD §5) is enforced server-side on every list and detail endpoint. The filters documented in each endpoint's Parameters section are additional filters a user may apply on top of their implicit scope — they cannot widen the scope beyond what the caller's role permits. A Team Lead passing another team's pod ID as a filter parameter receives an empty result, not a 403, so the frontend does not need to manage scope visibility — it can simply display what the API returns.

12. Webhooks & Async Event Reference
The platform does not expose outbound webhooks to external consumers in Phase 1–3. Internally, Celery jobs fire on schedule or on-demand; the frontend polls job-status endpoints (Section 10) for async operations. This section documents the internal async events that trigger side effects — notification dispatches, cache invalidations and audit entries — so the backend Developer has a complete picture of what happens beyond the immediate HTTP response.
